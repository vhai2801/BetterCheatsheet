# Better Cheatsheet

## Overview
A native macOS menu-bar/window app for keeping notes on custom keyboard shortcuts. Notes are organized into user-named tabs (unlimited count, starts empty — no default tab) shown in a top tab bar with a visible border per tab. Every tab's format is fixed at creation and can never be flipped afterward: the "+" control always creates a templated two-column Shortcut/Action table (`ShortcutTableView`), while a separate note-icon button creates a freeform rich-text "Note tab" instead. A global hotkey (user-configurable, default **⇧⌘K**) toggles a floating overlay panel that shows the same tabs/content as a quick-glance cheatsheet, without switching away from whatever app you're using; an ellipsis ("...") button in the overlay's tab bar jumps to the main editor window and closes the overlay. Note tabs can individually be marked "editable in overlay" so you can jot things down without opening the full editor window; table tabs are always read-only in the overlay and only editable in the main window. Note-tab content is rich text: a small toolbar (Bold, font size −/+, Font Panel) lets you format it. In the Shortcut column of a table tab, holding a modifier (⌘⌥⇧⌃) or pressing Space/Return/an arrow key inserts its symbol the instant it's physically pressed; Tab does too, but a single press inserts "⇥" while two presses in quick succession do a real Tab (move to the Action field) instead - see `ShortcutTableView.swift`. (The old ALL-CAPS-keyword-while-typing auto-replace, e.g. typing `CMD` to get "⌘", was removed 2026-07-07 - physical key capture supersedes it; see Decisions log.) A gear icon pinned to the tab bar's trailing edge (always visible, not part of the scrollable tab strip) opens Settings, where you can re-record the global shortcut, optionally make left/right modifier side matter (e.g. only Right-Shift, not either Shift), and pick a Light/Dark/Frosted-Glass theme — Frosted Glass only affects the overlay; the main window always stays opaque. Tabs can be dragged to reorder them. The app has a minimal `NSApp.mainMenu` (Quit, Close, Minimize) so Cmd+W/Cmd+Q/Cmd+M work despite there being no NIB; closing the main window (Cmd+W or the traffic light) drops the app back to a Dock-less menu-bar-only presence until it's reopened. The app makes no network connections and stores notes locally only, at `~/Library/Application Support/BetterCheatsheet/tabs.json` (settings alongside it in `settings.json`).

Built as a Swift Package (not an Xcode project) because only Xcode Command Line Tools are installed, not full Xcode. `build.sh` wraps the compiled binary into a real, ad-hoc-codesigned `BetterCheatsheet.app` bundle.

## Architecture map
- `Package.swift` — executable target `BetterCheatsheet`, macOS 13+, zero dependencies
- `Sources/BetterCheatsheet/main.swift` — entry point: creates `NSApplication`, sets `.regular` activation policy, assigns `AppDelegate`, calls `app.run()`
- `Sources/BetterCheatsheet/AppDelegate.swift` — conforms to `NSWindowDelegate` too now (for `windowWillClose`, see below). Owns the main editor `NSWindow` (default size 760x560), the floating `OverlayPanel` (an `NSPanel` subclass with `.nonactivatingPanel` + `.resizable` style, deliberately **not** `.titled` — borderless, so there's no reserved title-bar strip at all rather than one with hidden traffic lights; square corners from being borderless are rounded again by `CheatsheetView`'s own `.clipShape`; stays permanently non-opaque/clear (`isOpaque = false`, `backgroundColor = .clear`) independent of theme so that clip actually shows through), the `NSStatusItem` menu bar icon/menu (Show Cheatsheet / Edit Notes / Quit), and registers the global hotkey. `registerHotKey(_:)` dispatches between the Carbon `HotKeyManager` and `SideSensitiveHotKeyMonitor` based on `hotKey.sideSensitive`, tearing down whichever isn't in use. `setUpMainMenu()` builds a minimal `NSApp.mainMenu` (Quit under the app menu; Close/Minimize under a Window menu) — needed because a bare Swift executable with no NIB gets no menu bar at all from AppKit, so without this Cmd+W/Cmd+M silently do nothing. `windowWillClose(_:)` (the `NSWindowDelegate` method) sets `NSApp.setActivationPolicy(.accessory)` when the *main* window closes, dropping the Dock icon since the app is meant to live in the menu bar the rest of the time; `applicationShouldHandleReopen` restores `.regular` + reopens the editor on a Dock-icon click or relaunch. The overlay's `onOpenMainWindow` closure (wired to `CheatsheetView`'s "..." button) orders the overlay out and calls the same `showEditor()` the status-bar menu uses. Remembers the overlay's position/size via `setFrameAutosaveName("BetterCheatsheetOverlayFrame")`/`setFrameUsingName`, only auto-centering near the cursor the very first time it's ever shown — `overlayHasPlacement` tracks this. `observeSettings()` has a single `settings.$hotKey.dropFirst().sink` doing both `registerHotKey` and `updateStatusItemTitle` (not two separate subscriptions - the initial value is already handled directly by the setup calls that run before this subscribes). `applicationWillTerminate(_:)` calls `appState.flushPendingSave()` so a debounced save (see Models.swift) still lands before the process exits
- `Sources/BetterCheatsheet/HotKeyManager.swift` — wraps Carbon's `RegisterEventHotKey`/`InstallEventHandler`. Chosen over a `CGEventTap` specifically to avoid needing Accessibility/Input Monitoring permission on first launch
- `Sources/BetterCheatsheet/Models.swift` — `ShortcutRow` (Codable, plain `shortcut`/`action` strings, no rich text — the point of the table format is a consistent look, not per-row formatting) and `TabItem` (Codable; rich text stored as RTF `Data` in `rtfData` - defaults to a literal empty `Data()`, not an RTF-round-tripped empty string, since `decodeRTF` already special-cases `data.isEmpty` to skip parsing; exposed via computed `attributedContent: NSAttributedString`; custom `init(from:)` migrates tabs saved before rich text existed, which had a plain `content: String` field, and defaults `shortcutRows` to `[]` for tabs saved before the table format existed; RTF decode results are cached in an `NSCache` keyed by the raw bytes, `countLimit`-bounded since every edit produces a new byte blob - and so a new never-otherwise-evicted entry - rather than updating one in place). `editableInOverlay` now doubles as "is this tab a freeform Note tab" — true means rich-text (`AutoReplaceTextEditor`, editable in the overlay too); false means the `shortcutRows`-backed table (read-only in the overlay). `AppState` (ObservableObject; loads/saves `[TabItem]` as JSON to Application Support — persists selected tab id in `UserDefaults`) seeds a brand-new install (no `tabs.json` yet, or one that fails to decode) with `starterTabs()` - two pre-filled tabs, "General" (app/window management shortcuts) and "Texts based" (text-editing shortcuts), so a new user sees something useful immediately instead of a blank slate (added 2026-07-07 - **do not reintroduce a bare `tabs = []` fallback here**, that's the blank-slate regression this fixed). They're ordinary tabs from that point on, freely renamable/editable/deletable like any other - nothing tracks that they were auto-created. Two creation paths for new tabs: `addTab(named:)` (the "+" flow) makes a table tab pre-seeded with one empty `ShortcutRow` so there's already a field to type into; `addNoteTab()` makes a freeform tab named "Note" (or "Note 2", "Note 3", ... via `uniqueNoteTabName()`) with `editableInOverlay = true`. Neither path nor anything else ever flips `editableInOverlay` on an existing tab — that's deliberate, see Decisions log. `moveTab(id:toIndex:)` replaced the old `moveTab(id:before:)`, taking a target index in the *current* ordering (same convention as `List.onMove`) to match `TabBarView`'s rewritten gap-based drag-reorder. Saving is debounced: `tabs`'s `didSet` calls `scheduleSave()`, not a direct encode+write - every keystroke anywhere mutates `tabs`, so a direct synchronous write there was a full JSON-encode-and-atomic-write of *every* tab's content on every character typed, on the main thread (see Decisions log, fixed 2026-07-07). `scheduleSave()` cancels any pending `DispatchWorkItem`, snapshots `tabs` (a value-type copy, safe off-thread), and schedules the actual encode/write on a background `saveQueue` ~0.4s later; `flushPendingSave()` (called from `AppDelegate.applicationWillTerminate`) writes immediately and cancels the pending item, so a save scheduled just before quitting still lands
- `Sources/BetterCheatsheet/TextReplacement.swift` — now just `spelledOut(_:)`: symbol → spelled-out name (e.g. "⌘⇧K" → "Cmd Shift K"), used by the Settings "show shortcuts as text" toggle, via a `symbolNames` lookup table. Used to also hold the ALL-CAPS-keyword → symbol live-typing auto-replace (`map`/`replacement(in:beforeLocation:)`) - removed 2026-07-07, see Decisions log
- `Sources/BetterCheatsheet/AutoReplaceTextEditor.swift` — `NSViewRepresentable` wrapping a rich-text `NSTextView` (`isRichText = true`, `usesFontPanel = true`). Name is a holdover from when it also applied `TextReplacement`'s ALL-CAPS live auto-replace at the caret - that was removed 2026-07-07 (see Decisions log); not renamed since Note-tab rich text is still its whole job. `Coordinator.parent` is reassigned to `self` at the top of every `updateNSView` call — **do not remove this**: without it the coordinator keeps writing every keystroke into whichever tab's binding was current when it was first created, so any tab other than the first becomes silently untypable (real bug, fixed 2026-07-05). `Coordinator.isProgrammaticUpdate` guards `textDidChange` during our own `setAttributedString` calls (tab switches) — **do not remove this either**: `setAttributedString` triggers the same delegate callback a real edit would, so without the guard every tab switch wrote the content straight back into the model (full RTF encode + synchronous `tabs.json` rewrite on the main thread), which was the likely cause of a reported tab-switch lag (fixed 2026-07-05, not yet confirmed by the user)
- `Sources/BetterCheatsheet/TextFormattingController.swift` — plain class holding `weak var textView: NSTextView?`; toolbar buttons in `EditorView` call `toggleBold()`/`increaseFontSize()`/`decreaseFontSize()`/`showFontPanel()` on it, which apply to the current selection or (if none) `typingAttributes`. `AutoReplaceTextEditor` points its `.textView` at whichever NSTextView is currently mounted
- `Sources/BetterCheatsheet/TabBarView.swift` — horizontal scrolling tab bar; each `TabButton` has a visible 1pt border (accent-tinted when selected), collapses to `maxWidth: 110` and expands to fit content on hover (animated, but debounced 250ms via a cancellable `DispatchWorkItem` so scrolling the strip doesn't trigger expand on every tab the cursor crosses, and suspended entirely while a drag is in progress so the floating drag copy sweeping over neighbors doesn't fight the reorder animation), right-click for rename/delete. `TabButton`'s tap gesture is a single plain `.onTapGesture` (select) - **do not add a `count: 2` sibling gesture back** for double-click-to-rename: having both a count:1 and count:2 recognizer on the same view forces the system to wait out the double-click window before committing to the single-tap action, which made every tab click feel like it took about a second to register (real bug, fixed 2026-07-07). The context menu's "Rename" already covers renaming, so nothing is lost by not having a double-click shortcut for it too. Drag-to-reorder was rewritten from the original `.onDrag`/`.onDrop(delegate:)` approach (which only reacted once the cursor was precisely over another tab's own bounds) to a gap-based design: a thin `gapView` marker sits between every pair of tabs (and at both ends), normally zero-width/invisible, driven purely by the drag's horizontal x-position via `updateInsertionIndex(forX:draggingID:)` rather than by hover-testing other tabs — so it's always exactly at a real tab boundary regardless of individual tab widths, and dropping anywhere in the window (not just inside the tab bar) still commits to the nearest gap. Each tab's frame is tracked in `tabFrames` (a `[UUID: CGRect]` populated via `UUIDFramePreferenceKey<TabFrameTag>` - see FramePreferenceKey.swift - from a `GeometryReader` behind every tab, all sharing one named coordinate space `dragSpace`); the dragged tab itself stays mounted but collapses to zero width/opacity (`draggingTabID`) so its `highPriorityGesture` isn't torn down mid-drag, while a non-interactive `floatingTab` copy (positioned via `draggedTabOriginFrame` + `dragTranslation`, `.shadow`ed, `.allowsHitTesting(false)`) lives in an `.overlay` outside the `ScrollView` specifically so it isn't clipped to the tab strip's height. `commitReorder()` calls `AppState.moveTab(id:toIndex:)` on drag end. The add-tab control ("+"), a separate note-icon button (`addNoteTabButton`, creates a freeform Note tab — see Models.swift), and the Settings gear are all fixed *outside* the scrollable `HStack` (siblings, not part of the `ForEach`), so none can be pushed out of view by tab overflow — do not move any of them back inside the scroll view. The overlay additionally shows an ellipsis (`onOpenMainWindow`) button in this same fixed-outside group. The "+" flow's text field is styled like a real tab pill (grows with the typed name via `fixedSize`) rather than a plain box, and on submit calls `AppState.addTab(named:)` then sets `AppState.pendingContentFocusTabID` so the new tab's first table row grabs focus immediately (see EditorView/ShortcutTableView)
- `Sources/BetterCheatsheet/FramePreferenceKey.swift` — `UUIDFramePreferenceKey<Tag: FrameTag>`, a generic `[UUID: CGRect]`-merging `PreferenceKey`, replacing three structurally-identical bespoke types that used to live separately in ShortcutTableView.swift/TabBarView.swift. `Tag` (`RowFrameTag`/`GripFrameTag`/`TabFrameTag`, each an empty `enum` conforming to the marker protocol `FrameTag`) exists purely so distinct call sites get their own preference "channel" - SwiftUI's preference propagation is scoped per conforming *type*, not per usage site, so e.g. ShortcutTableView's row frames and its grip-handle frames (both keyed by the same row IDs) would silently merge into one dictionary and clobber each other if they shared one concrete type
- `Sources/BetterCheatsheet/EditorView.swift` — main window content: tab bar + a per-tab toolbar that shows either the rich-text controls (Bold/font size/Font Panel via `TextFormattingController`, Note tabs only) or "Add row" + font size −/+ buttons (table tabs only - "Add row" appends a `ShortcutRow`, the font buttons adjust `settings.shortcutTableFontSize` directly, clamped to 9...28), plus a trash-icon delete-tab button always present, then either `AutoReplaceTextEditor` (Note tabs) or `ShortcutTableView` (table tabs, `isEditable: true`) for the selected tab's content, or `SettingsView` when Settings is selected (`appState.isShowingSettings`). Which branch renders is driven by `tab.editableInOverlay` and is fixed at tab-creation time — there is no toggle here to flip a tab's format after the fact (see Models.swift/Decisions log). The `ShortcutTableView` instance is `.id(tab.id)`-tagged to force a fresh mount per tab, which is what makes its focus-on-appear behavior (see below) actually refire for every newly-created table tab rather than just the first one ever shown. No Frosted Glass background here at all anymore — the main window always stays opaque regardless of theme (see Decisions log)
- `Sources/BetterCheatsheet/CheatsheetView.swift` — overlay content: tab bar (no add controls, no settings icon, but an ellipsis `onOpenMainWindow` button — see TabBarView/AppDelegate) + either the same `AutoReplaceTextEditor` (Note tabs flagged `editableInOverlay`) or a read-only `ShortcutTableView(rows: .constant(tab.shortcutRows), isEditable: false)` (table tabs — always non-editable in the overlay regardless of `editableInOverlay`, which for table tabs is always false anyway). Root view uses a flexible `.frame(minWidth:maxWidth:minHeight:maxHeight:)` — **do not hardcode a fixed frame here again**: it previously had `.frame(width: 420, height: 360)`, which fought any attempt to resize the overlay panel (content just snapped back). Frosted Glass here keeps `.hudWindow`/`.behindWindow` (the true live desktop blur) since this is the small glanceable HUD where that look is the point
- `Sources/BetterCheatsheet/ShortcutTableView.swift` — the templated two-column (Shortcut, Action) table shown for any tab that isn't a Note tab, built with SwiftUI `Grid` (not a fixed layout). Each editable cell is `ShortcutTableTextField`, a plain `NSTextField`-wrapping `NSViewRepresentable` (SwiftUI's own `TextField` on macOS 13 can't intercept individual keystrokes or Tab). In the Shortcut column only (`capturesModifierKeys`), its `Coordinator` inserts a modifier's own symbol the instant it's physically pressed (hold Shift+Cmd+K to type "⇧⌘K" directly, via a local `flagsChanged` monitor started only while that field is being edited), plus does the same live-symbol treatment for Space, Return, and the arrow keys (via `doCommandBy:`/a local `keyDown` monitor for Space specifically, since Space isn't a modifier and never reaches `flagsChanged`) - tied to `capturesModifierKeys` so the Action column (which never sets it) keeps normal text-field behavior throughout. `handleFlagsChanged` also special-cases a Hyper key (Caps Lock or similar remapped via Karabiner/Hammerspoon to emit Ctrl+Opt+Cmd+Shift at once): if all four modifiers turn on together in a single `flagsChanged` event (`pressed == hyperFlags`) - the tell that distinguishes a Hyper key from someone actually holding four physical keys down one at a time, which fires one event per key - it inserts the literal word `"Hyper "` instead of stacking the four individual symbols. Fn is handled similarly but standalone (no combo to detect) - it has no single glyph the way ⌘⇧⌥⌃ do, so a fresh `.function` press inserts `"fn "`. `TextReplacement.spelledOut(_:)` knows to match and preserve both `"Hyper"` and `"fn"` (its `literalTokens` list) as single tokens rather than exploding them letter-by-letter, since (unlike every other captured key) they're multiple characters, not a single symbol glyph. Tab is special-cased further: `handleTab` schedules the "⇥" symbol on a `DispatchWorkItem` after `tabDoubleTapWindow` (0.35s); a second Tab press arriving before that fires cancels the pending symbol and does real Tab navigation instead (`NSWindow.selectKeyView(following:)`/`selectKeyView(preceding:)` for Shift+Tab - **not** `selectNextKeyView`/`selectPreviousKeyView`, which resolve relative to the window's current first responder, i.e. the transient field editor rather than the actual `NSTextField`, and so silently no-op). On the Action column, plain `doCommandBy:` interception is still used, but only for `onTab` on the last row's Action field, to append a new `ShortcutRow` and focus it instead of tabbing off into nothing. `focusFirstRowOnAppear` (set by `EditorView` from `AppState.pendingContentFocusTabID`) jumps focus straight into the new tab's first Shortcut field right after naming it; applied both `onAppear` and `onChange(of: focusFirstRowOnAppear)` since the three separate `@Published` writes behind it don't always land in the same SwiftUI update cycle. Read-only mode (overlay) swaps every cell for plain selectable `Text`. Both the `NSTextField` (editable) and `Text` (read-only) cells render at `settings.shortcutTableFontSize` - an adjustable, persisted font size (+/- buttons in `EditorView`'s per-tab toolbar), shared between the main window and overlay the same way the column width/inset are. The read-only Shortcut `Text` additionally respects `settings.shortcutsDisplayAsText`, passing the stored symbol string through `TextReplacement.spelledOut(_:)` when on - the editable `NSTextField` (main window) does not, staying symbol-based always (see Decisions log for why this is scoped to read-only display only).
  - **Scrolls both axes, not just vertical**: the Shortcut column has a user-configurable fixed width and the Action column has a floor (`actionColumnMinWidth`) - their combined minimum can exceed a narrow window/overlay's actual width, and a vertical-only `ScrollView` still constrains its content to its own width in the cross axis regardless of the Grid's `.fixedSize`, so that case was silently collapsing/clipping the Shortcut column instead of scrolling (reported as "text doesn't move properly / cuts off text when resizing the overlay, even though there's clearly enough space" - there wasn't enough space for the *configured minimums*; fixed 2026-07-07 by adding `.horizontal` to the `ScrollView`'s axes). The content's `.frame` is `minWidth`/`minHeight` pinned to the ScrollView's own last-measured size (`scrollViewSize`, a `CGSize` tracked via `background`+`GeometryReader`+`ScrollViewSizePreferenceKey` - not just a width, despite the name of the computed `scrollViewWidth` property still used by `actionColumnWidth`/the separator overlay), **not** `maxWidth`/`maxHeight: .infinity` - a doubly-scrollable `ScrollView` proposes an unbounded/intrinsic size to its content along *both* axes, so `.infinity` had no concrete offered size to grow into and did nothing; a concrete minimum the ScrollView can't out-propose is what actually keeps a wide/tall viewport filled and a narrow/short one scrolling instead of centering-or-clipping its content. The `Grid` itself is `.fixedSize` in **both** axes (not just horizontal) for the same reason: once that outer minHeight floor actually started working, a Grid that was still vertically flexible took the new tall proposed height as license to stretch its own rows to fill it - and since the Shortcut column header's own `.frame(alignment: .leading)` implies vertical *center* (not top) for that one cell, its text rendered centered within an inflated row while the plain "Action" `Text` stayed pinned to the top, visibly splitting the two header labels apart on any short-content tab, e.g. a freshly-created one (real bug, fixed 2026-07-07 - **do not drop `vertical: true` from the Grid's `.fixedSize` again**, that's the regression this fixed).
  - **Shortcut column resizing**: two drag handles at the header only (`shortcutColumnCell(showsHandles:)`) - a leading one that pushes the text further in (`shortcutColumnLeadingInset`) without changing the column's width, and a trailing one (at the Shortcut/Action boundary) that changes the width directly (`shortcutColumnWidth`). Both values live in `SettingsStore`/`settings.json`, not local `@State`, because `EditorView` force-remounts this view per tab (`.id(tab.id)`) and the main window plus the overlay are two separate instances that must agree on layout - handles only render when `isEditable` (main window), but the overlay reads and reflects the same persisted values. Both handles track the drag's translation in the shared named coordinate space (`rowDragSpace`, see below), not their own local frame - the trailing handle's own position shifts as a side effect of the very drag it's tracking, so a local-frame translation fed back into itself and read as stuck/jittery. Rows reserve the same leading width the handle occupies (`Color.clear.frame(width: 9)` in place of `leadingInsetHandle`) so "Shortcut" and every row's actual value start at the same x position - without that spacer, the header text alone sat further right than the rows (real bug, fixed 2026-07-07); the leading handle's gap to "Shortcut" and the trailing handle's gap to "Action" both use spacing `12` to match the Grid's own `horizontalSpacing` (they'd previously drifted to `4` vs `12` and looked uneven, also fixed 2026-07-07).
  - **Row drag-to-reorder**: same gap-based design as `TabBarView`'s tab reordering, adapted to vertical dragging - a small grip handle (`gripHandle(for:)`) per row carries the `DragGesture` so it doesn't fight the text fields' own click-to-place-cursor behavior; `rowFrames`/`gripFrames` (`[UUID: CGRect]`, via `UUIDFramePreferenceKey<RowFrameTag>`/`<GripFrameTag>` - see FramePreferenceKey.swift - in the shared `rowDragSpace`) locate each row/handle, `updateInsertionIndex(forY:draggingID:)` finds which gap the drag's y-position falls in, `commitReorder()` calls `moveRow(id:toIndex:)` on release. The dragged row stays mounted but collapsed to zero height/opacity so its gesture isn't torn down mid-drag; a floating badge shows just that row's Shortcut text, anchored to the grip handle's own tracked frame (not centered across the whole, potentially Action-widened, row - centering there made the preview jump toward the row's far end).
  - **Full-width separator lines**: resting divider lines (below the header, between every row) are drawn in a single `separatorOverlay`, positioned via `headerFrame`/`rowFrames` and spanning the full tracked `scrollViewWidth` - deliberately *not* drawn inside the `Grid` itself, because a full-bleed-seeking line inside a Grid cell forces Grid to inflate whichever column track it spans (the same mechanism that was inflating the Shortcut column before `.fixedSize` was added). `scrollViewWidth` comes from a `background`+`GeometryReader`+`PreferenceKey` on the `ScrollView` itself, not a `GeometryReader` placed directly as the overlay's own content - the latter measures the (deliberately narrow, `.fixedSize`) `Grid`'s own natural width instead of the true viewport, which is also why the native scrollbar used to stop short of the window's true trailing edge (the `ScrollView`'s content now gets its own `.frame(maxWidth: .infinity, alignment: .leading)` wrapping the fixed-size `Grid`, so the scroller/document view is genuinely as wide as the window while the `Grid`'s columns stay exactly as configured). **Known `Grid` quirk that bit this twice**: modifiers chained *after* a `GridRow`'s closure are broadcast to every cell in that row individually, not applied to the row as a single unit - the header's frame-measuring `GeometryReader` must be attached directly to one specific cell (`Text("Action")`), not chained onto the whole `GridRow`, or every cell (including a 1pt-tall spacer) writes its own competing preference value and whichever is processed last wins (this placed the separator across the middle of the header text instead of below it - fixed 2026-07-07).
  - **Action column width**: computed (`actionColumnWidth`), not just a `minWidth` - derived from `scrollViewWidth` minus every other fixed-width neighbor (grip column, Shortcut column + its handle, delete button, Grid spacings, outer padding) so it stretches to fill whatever space is actually available, and the per-row delete (`minus.circle`) button ends up pinned near the window's trailing edge like the native scrollbar, instead of sitting wherever Action's own content happened to end (fixed 2026-07-07). Grid sizes a column to the widest cell in it across every row, so the header's own "Action" text is left unconstrained and just renders left-aligned within whatever width the data rows establish.
- `Sources/BetterCheatsheet/Settings.swift` — `AppTheme` enum (`light`/`dark`/`frostedGlass`), `HotKeyConfig` struct (`keyCode`, `modifiers` (generic Carbon mask), `modifierKeyCodes` (specific physical keys recorded, for side-sensitive matching), `sideSensitive`; custom Codable `init(from:)` migrates configs saved before the last two fields existed), `SettingsStore` (ObservableObject, persists both to `settings.json`, plus `shortcutColumnWidth`/`shortcutColumnLeadingInset`/`shortcutTableFontSize`/`shortcutsDisplayAsText` for `ShortcutTableView` - all four shared between the main window and overlay for the same reason: `EditorView` force-remounts `ShortcutTableView` per tab, and the two windows are separate instances that must agree on layout). `shortcutsDisplayAsText` is read-only-display-only (see ShortcutTableView.swift/TextReplacement.swift entries and Decisions log) - toggled via `SettingsView`'s "Shortcut Display" section
- `Sources/BetterCheatsheet/ModifierKeyCode.swift` — pure logic mapping physical left/right modifier virtual keyCodes (`kVK_Command` vs `kVK_RightCommand` etc.) to their semantic `NSEvent.ModifierFlags` category, plus display symbol/side and `defaultLeftKeyCodes(forCarbonModifiers:)` (used when turning on side-sensitivity for a hotkey that's never captured specific sides)
- `Sources/BetterCheatsheet/AccessibilityPermission.swift` — thin wrapper around `AXIsProcessTrustedWithOptions` to prompt for Accessibility permission, needed only for side-sensitive hotkey matching
- `Sources/BetterCheatsheet/CapsLockSuppressor.swift` — `forceOff()` force-clears the OS's native Caps Lock lock-state via `IOHIDSetModifierLockState` against the `IOHIDSystem` service (the same trick dedicated "disable Caps Lock" utilities use), for a Caps Lock key that's remapped elsewhere (e.g. to Hyper) but still also engages the native lock alongside the remap. A plain IOKit call, not a `CGEventTap` - needs no Accessibility permission. Called from `ShortcutTableView`'s `handleFlagsChanged` only while a Shortcut cell has focus, not app-wide
- `Sources/BetterCheatsheet/SideSensitiveHotKeyMonitor.swift` — alternative to `HotKeyManager` for when modifier side matters: global+local `NSEvent` monitors for `.flagsChanged` (tracks currently-held modifier keyCodes) and `.keyDown` (checks against the recorded set). Requires Accessibility permission to receive events from other apps at all - Carbon's `RegisterEventHotKey` fundamentally cannot distinguish left/right, so this is a wholesale alternate path, not an enhancement to it
- `Sources/BetterCheatsheet/HotKeyFormatter.swift` — pure logic: `HotKeyConfig` → display string (e.g. "⇧⌘K", or "R⇧L⌘K" when side-sensitive), resolves the base character for a keyCode via `UCKeyTranslate` so it's correct regardless of keyboard layout. `character(forKeyCode:)` memoizes results per keyCode in `characterCache` (main-thread only, no locking) - the underlying Carbon `TISCopyCurrentKeyboardLayoutInputSource`+`UCKeyTranslate` calls aren't free and the keyboard layout essentially never changes mid-session, but this was getting re-run on every call including from SettingsView's body re-evaluating `string(for:)` on every render
- `Sources/BetterCheatsheet/HotKeyRecorderView.swift` — click-to-record `NSViewRepresentable`; captures the next keyDown with at least one modifier held, Esc cancels. Always tracks specific held modifier keyCodes via `flagsChanged` regardless of whether side-sensitivity is on, so enabling it later doesn't require re-recording
- `Sources/BetterCheatsheet/VisualEffectBackground.swift` — `NSVisualEffectView` wrapper, `blendingMode` configurable per call site (see EditorView vs CheatsheetView above); the hosting window must be non-opaque with a clear background color for `.behindWindow` blending to actually show the desktop through it
- `Info.plist` — app bundle metadata (`LSMinimumSystemVersion` 13.0, no network entitlements requested, `CFBundleIconFile: AppIcon`)

## Release process (Homebrew cask distribution)
The app is distributed as a Homebrew cask so the user can install/update it from the terminal on any machine, without needing this repo cloned there. **This is the runbook to follow whenever asked to "update the app and push a new version."**

**Where things live:**
- Main repo: `https://github.com/vhai2801/BetterCheatsheet` (public — was private, made public specifically so release assets are downloadable by `brew install` with zero auth on any machine)
- Tap repo: `https://github.com/vhai2801/homebrew-better-cheatsheet` (public), local clone at `~/Projects/homebrew-better-cheatsheet` on this machine — **on a different machine, `git clone https://github.com/vhai2801/homebrew-better-cheatsheet.git` first**, it won't already be there
- Cask file: `Casks/better-cheatsheet.rb` in the tap repo
- User installs with: `brew tap vhai2801/better-cheatsheet && brew install --cask better-cheatsheet`
- User updates with: `brew update && brew upgrade --cask better-cheatsheet`

**To cut a new release (vX.Y.Z), in order:**
1. In `~/Projects/BetterCheatsheet`, bump `CFBundleShortVersionString` in `Info.plist` to the new version (semver, matches the git tag and cask `version` — keep all three in sync)
2. `./build.sh --release` (release config, not the debug config used during day-to-day dev)
3. Package it: `ditto -c -k --keepParent .build/arm64-apple-macosx/release/BetterCheatsheet.app BetterCheatsheet-X.Y.Z.zip` — **use `ditto`, not `zip -r`**, it's the macOS-safe way to zip an `.app` bundle without mangling it
4. `shasum -a 256 BetterCheatsheet-X.Y.Z.zip` — save this hash, the cask needs it
5. Commit the version bump, push, then tag and push the tag: `git tag vX.Y.Z && git push origin vX.Y.Z`
6. `gh release create vX.Y.Z BetterCheatsheet-X.Y.Z.zip --title "vX.Y.Z" --notes "..."` — attaches the zip as a release asset
7. Verify the asset is actually publicly fetchable: `curl -sL -o /dev/null -w "%{http_code}\n" "https://github.com/vhai2801/BetterCheatsheet/releases/download/vX.Y.Z/BetterCheatsheet-X.Y.Z.zip"` should print `200`
8. In `~/Projects/homebrew-better-cheatsheet`, edit `Casks/better-cheatsheet.rb`: update `version` and `sha256` to the new values (the `url` templates `#{version}` in automatically, no need to touch it)
9. Commit and push the cask repo

After step 9, `brew update && brew upgrade --cask better-cheatsheet` on any machine (including this one) will pick up the new version. Test with `brew upgrade --cask better-cheatsheet` locally to confirm before considering the release done.

**Things that will bite you if forgotten:**
- The cask requires `depends_on macos: :ventura` (bare symbol) — the `">= :ventura"` string comparison form is deprecated and prints a warning on every tap/install
- The very first time a machine taps this repo, Homebrew's tap-trust security feature blocks loading the cask until `brew trust vhai2801/better-cheatsheet` is run once — if `brew install` fails with "Refusing to load cask ... from untrusted tap," that's why
- The app isn't notarized (no paid Apple Developer ID, only ad-hoc `codesign --sign -`), so downloaded copies get a quarantine flag that would trigger a Gatekeeper "unidentified developer" prompt on first launch. The cask's `postflight` block strips it via `xattr -dr com.apple.quarantine` — **do not remove this block**, or every fresh install needs a manual right-click-Open the first time
- Keep `CFBundleShortVersionString`, the git tag, and the cask's `version` field all identical (e.g. all `1.0.0`, tag `v1.0.0`) - the cask's `url` is built from `#{version}` and will 404 if they drift
