# Better Cheatsheet

## Overview
A native macOS menu-bar/window app for keeping notes on custom keyboard shortcuts. Notes are organized into user-named tabs (unlimited count, starts empty — no default tab) shown in a top tab bar with a visible border per tab. A global hotkey (user-configurable, default **⇧⌘K**) toggles a floating overlay panel that shows the same tabs/content as a quick-glance cheatsheet, without switching away from whatever app you're using. Tabs can individually be marked "editable in overlay" so you can jot things down without opening the full editor window. Content is rich text: a small toolbar (Bold, font size −/+, Font Panel) lets you format it. While editing, typing an exact ALL-CAPS modifier keyword (e.g. `CMD`, `OPTION`, `SHIFT`) auto-replaces it with its symbol (⌘, ⌥, ⇧, ...) as soon as you type a word-boundary character after it. A gear icon pinned to the tab bar's trailing edge (always visible, not part of the scrollable tab strip) opens Settings, where you can re-record the global shortcut, optionally make left/right modifier side matter (e.g. only Right-Shift, not either Shift), and pick a Light/Dark/Frosted-Glass theme — Frosted Glass only affects the overlay; the main window always stays opaque. Tabs can be dragged to reorder them. The app makes no network connections and stores notes locally only, at `~/Library/Application Support/BetterCheatsheet/tabs.json` (settings alongside it in `settings.json`).

Built as a Swift Package (not an Xcode project) because only Xcode Command Line Tools are installed, not full Xcode. `build.sh` wraps the compiled binary into a real, ad-hoc-codesigned `BetterCheatsheet.app` bundle.

## Architecture map
- `Package.swift` — executable target `BetterCheatsheet`, macOS 13+, zero dependencies
- `Sources/BetterCheatsheet/main.swift` — entry point: creates `NSApplication`, sets `.regular` activation policy, assigns `AppDelegate`, calls `app.run()`
- `Sources/BetterCheatsheet/AppDelegate.swift` — owns the main editor `NSWindow` (default size 760x560), the floating `OverlayPanel` (an `NSPanel` subclass with `.nonactivatingPanel` style so it can become key for typing without stealing app focus; `.resizable`; traffic-light buttons explicitly hidden via `standardWindowButton(...)?.isHidden = true` since hiding the title text alone doesn't remove them; remembers its position/size via `setFrameAutosaveName("BetterCheatsheetOverlayFrame")`/`setFrameUsingName`, only auto-centering near the cursor the very first time it's ever shown — `overlayHasPlacement` tracks this), the `NSStatusItem` menu bar icon/menu (Show Cheatsheet / Edit Notes / Quit), and registers the global hotkey. `registerHotKey(_:)` dispatches between the Carbon `HotKeyManager` and `SideSensitiveHotKeyMonitor` based on `hotKey.sideSensitive`, tearing down whichever isn't in use
- `Sources/BetterCheatsheet/HotKeyManager.swift` — wraps Carbon's `RegisterEventHotKey`/`InstallEventHandler`. Chosen over a `CGEventTap` specifically to avoid needing Accessibility/Input Monitoring permission on first launch
- `Sources/BetterCheatsheet/Models.swift` — `TabItem` (Codable; rich text stored as RTF `Data` in `rtfData`, exposed via computed `attributedContent: NSAttributedString`; custom `init(from:)` migrates tabs saved before rich text existed, which had a plain `content: String` field; RTF decode results are cached in an `NSCache` keyed by the raw bytes since SwiftUI re-evaluates binding getters on every body pass) and `AppState` (ObservableObject; loads/saves `[TabItem]` as JSON to Application Support — starts as `[]`, no default tab — persists selected tab id in `UserDefaults`)
- `Sources/BetterCheatsheet/TextReplacement.swift` — pure dictionary (ALL-CAPS keyword → symbol) + `TextReplacement.replacement(in:beforeLocation:)`, a pure NSString range-finder with no UIKit/AppKit dependency, easy to unit-test/extend
- `Sources/BetterCheatsheet/AutoReplaceTextEditor.swift` — `NSViewRepresentable` wrapping a rich-text `NSTextView` (`isRichText = true`, `usesFontPanel = true`); intercepts `shouldChangeTextIn:replacementString:` to apply `TextReplacement` live at the caret, preserving cursor position and typing attributes (so bold/font carries through the substitution). `Coordinator.parent` is reassigned to `self` at the top of every `updateNSView` call — **do not remove this**: without it the coordinator keeps writing every keystroke into whichever tab's binding was current when it was first created, so any tab other than the first becomes silently untypable (real bug, fixed 2026-07-05). `Coordinator.isProgrammaticUpdate` guards `textDidChange` during our own `setAttributedString` calls (tab switches) — **do not remove this either**: `setAttributedString` triggers the same delegate callback a real edit would, so without the guard every tab switch wrote the content straight back into the model (full RTF encode + synchronous `tabs.json` rewrite on the main thread), which was the likely cause of a reported tab-switch lag (fixed 2026-07-05, not yet confirmed by the user)
- `Sources/BetterCheatsheet/TextFormattingController.swift` — plain class holding `weak var textView: NSTextView?`; toolbar buttons in `EditorView` call `toggleBold()`/`increaseFontSize()`/`decreaseFontSize()`/`showFontPanel()` on it, which apply to the current selection or (if none) `typingAttributes`. `AutoReplaceTextEditor` points its `.textView` at whichever NSTextView is currently mounted
- `Sources/BetterCheatsheet/TabBarView.swift` — horizontal scrolling tab bar; each `TabButton` has a visible 1pt border (accent-tinted when selected, thicker + full accent when it's the current drag-drop target), collapses to `maxWidth: 110` and expands to fit content on hover (animated, but debounced 250ms via a cancellable `DispatchWorkItem` so scrolling the strip doesn't trigger expand on every tab the cursor crosses), draggable via `.onDrag`/`.onDrop(delegate: TabDropDelegate)` to reorder (`AppState.moveTab(id:before:)`), double-click to rename, right-click for rename/delete. The add-tab control ("+") and the Settings gear are both fixed *outside* the scrollable `HStack` (siblings, not part of the `ForEach`), so neither can be pushed out of view by tab overflow and adding a tab never requires scrolling — do not move either back inside the scroll view
- `Sources/BetterCheatsheet/EditorView.swift` — main window content: tab bar + formatting toolbar (Bold/font size/Font Panel via `TextFormattingController`) + "editable in overlay" checkbox + trash-icon delete-tab button + `AutoReplaceTextEditor` for the selected tab, or `SettingsView` when Settings is selected (`appState.isShowingSettings`). No Frosted Glass background here at all anymore — the main window always stays opaque regardless of theme (see Decisions log)
- `Sources/BetterCheatsheet/CheatsheetView.swift` — overlay content: tab bar (no add control, no settings icon) + read-only rich `Text(AttributedString(tab.attributedContent))` unless the tab is `editableInOverlay`, in which case the same `AutoReplaceTextEditor` is used. Root view uses a flexible `.frame(minWidth:maxWidth:minHeight:maxHeight:)` — **do not hardcode a fixed frame here again**: it previously had `.frame(width: 420, height: 360)`, which fought any attempt to resize the overlay panel (content just snapped back). Frosted Glass here keeps `.hudWindow`/`.behindWindow` (the true live desktop blur) since this is the small glanceable HUD where that look is the point
- `Sources/BetterCheatsheet/Settings.swift` — `AppTheme` enum (`light`/`dark`/`frostedGlass`), `HotKeyConfig` struct (`keyCode`, `modifiers` (generic Carbon mask), `modifierKeyCodes` (specific physical keys recorded, for side-sensitive matching), `sideSensitive`; custom Codable `init(from:)` migrates configs saved before the last two fields existed), `SettingsStore` (ObservableObject, persists both to `settings.json`)
- `Sources/BetterCheatsheet/ModifierKeyCode.swift` — pure logic mapping physical left/right modifier virtual keyCodes (`kVK_Command` vs `kVK_RightCommand` etc.) to their semantic `NSEvent.ModifierFlags` category, plus display symbol/side and `defaultLeftKeyCodes(forCarbonModifiers:)` (used when turning on side-sensitivity for a hotkey that's never captured specific sides)
- `Sources/BetterCheatsheet/AccessibilityPermission.swift` — thin wrapper around `AXIsProcessTrustedWithOptions` to prompt for Accessibility permission, needed only for side-sensitive hotkey matching
- `Sources/BetterCheatsheet/SideSensitiveHotKeyMonitor.swift` — alternative to `HotKeyManager` for when modifier side matters: global+local `NSEvent` monitors for `.flagsChanged` (tracks currently-held modifier keyCodes) and `.keyDown` (checks against the recorded set). Requires Accessibility permission to receive events from other apps at all - Carbon's `RegisterEventHotKey` fundamentally cannot distinguish left/right, so this is a wholesale alternate path, not an enhancement to it
- `Sources/BetterCheatsheet/HotKeyFormatter.swift` — pure logic: `HotKeyConfig` → display string (e.g. "⇧⌘K", or "R⇧L⌘K" when side-sensitive), resolves the base character for a keyCode via `UCKeyTranslate` so it's correct regardless of keyboard layout
- `Sources/BetterCheatsheet/HotKeyRecorderView.swift` — click-to-record `NSViewRepresentable`; captures the next keyDown with at least one modifier held, Esc cancels. Always tracks specific held modifier keyCodes via `flagsChanged` regardless of whether side-sensitivity is on, so enabling it later doesn't require re-recording
- `Sources/BetterCheatsheet/VisualEffectBackground.swift` — `NSVisualEffectView` wrapper, `blendingMode` configurable per call site (see EditorView vs CheatsheetView above); the hosting window must be non-opaque with a clear background color for `.behindWindow` blending to actually show the desktop through it
- `Info.plist` — app bundle metadata (`LSMinimumSystemVersion` 13.0, no network entitlements requested, `CFBundleIconFile: AppIcon`)
- `Resources/AppIcon.icns` — white ⌘ glyph on a purple-to-blue gradient squircle. No Xcode asset catalog available, so this was drawn programmatically (an AppKit/Core Graphics script, not checked into the repo - just the output) and compiled via `sips`/`iconutil`. Regenerate at 1024x1024 if it ever needs to change, then re-run through the same `sips` (resize to each required size) + `iconutil -c icns` pipeline
- `build.sh` — `swift build` + manual `.app` bundle assembly (including copying `Resources/AppIcon.icns` into `Contents/Resources/`) + ad-hoc `codesign`

## Release process (Homebrew cask distribution)
The app is distributed as a Homebrew cask so the user can install/update it from the terminal on any machine, without needing this repo cloned there. **This is the runbook to follow whenever asked to "update the app and push a new version."**

**Where things live:**
- Main repo: `https://github.com/vhai2801/BetterCheatsheet` (public — was private, made public specifically so release assets are downloadable by `brew install` with zero auth on any machine)
- Tap repo: `https://github.com/vhai2801/homebrew-better-cheatsheet` (public), local clone at `~/Projects/homebrew-better-cheatsheet` on this machine — **on a different machine, `git clone https://github.com/vhai2801/homebrew-better-cheatsheet.git` first**, it won't already be there
- Cask file: `Casks/better-cheatsheet.rb` in the tap repo
- User installs with: `brew tap vhai2801/better-cheatsheet && brew install --cask better-cheatsheet`
- User updates with: `brew update && brew upgrade --cask better-cheatsheet`

**To cut a new release (vX.Y.Z), in order:**
1. In `~/Projects/BetterCheatsheet`, bump `CFBundleShortVersionString` in `Info.plist` to the new version (semver, matches the git tag and cask `version` — keep all three in sync)
2. `./build.sh --release` (release config, not the debug config used during day-to-day dev)
3. Package it: `ditto -c -k --keepParent .build/arm64-apple-macosx/release/BetterCheatsheet.app BetterCheatsheet-X.Y.Z.zip` — **use `ditto`, not `zip -r`**, it's the macOS-safe way to zip an `.app` bundle without mangling it
4. `shasum -a 256 BetterCheatsheet-X.Y.Z.zip` — save this hash, the cask needs it
5. Commit the version bump, push, then tag and push the tag: `git tag vX.Y.Z && git push origin vX.Y.Z`
6. `gh release create vX.Y.Z BetterCheatsheet-X.Y.Z.zip --title "vX.Y.Z" --notes "..."` — attaches the zip as a release asset
7. Verify the asset is actually publicly fetchable: `curl -sL -o /dev/null -w "%{http_code}\n" "https://github.com/vhai2801/BetterCheatsheet/releases/download/vX.Y.Z/BetterCheatsheet-X.Y.Z.zip"` should print `200`
8. In `~/Projects/homebrew-better-cheatsheet`, edit `Casks/better-cheatsheet.rb`: update `version` and `sha256` to the new values (the `url` templates `#{version}` in automatically, no need to touch it)
9. Commit and push the cask repo

After step 9, `brew update && brew upgrade --cask better-cheatsheet` on any machine (including this one) will pick up the new version. Test with `brew upgrade --cask better-cheatsheet` locally to confirm before considering the release done.

**Things that will bite you if forgotten:**
- The cask requires `depends_on macos: :ventura` (bare symbol) — the `">= :ventura"` string comparison form is deprecated and prints a warning on every tap/install
- The very first time a machine taps this repo, Homebrew's tap-trust security feature blocks loading the cask until `brew trust vhai2801/better-cheatsheet` is run once — if `brew install` fails with "Refusing to load cask ... from untrusted tap," that's why
- The app isn't notarized (no paid Apple Developer ID, only ad-hoc `codesign --sign -`), so downloaded copies get a quarantine flag that would trigger a Gatekeeper "unidentified developer" prompt on first launch. The cask's `postflight` block strips it via `xattr -dr com.apple.quarantine` — **do not remove this block**, or every fresh install needs a manual right-click-Open the first time
- Keep `CFBundleShortVersionString`, the git tag, and the cask's `version` field all identical (e.g. all `1.0.0`, tag `v1.0.0`) - the cask's `url` is built from `#{version}` and will 404 if they drift

## Decisions log
- 2026-07-05: Chose Swift Package Manager + a manual `.app`-bundle build script over an Xcode project, because only Command Line Tools are installed (no `xcodebuild`). Revisit if the user installs full Xcode and wants project-file conveniences (asset catalogs for a real app icon, etc.).
- 2026-07-05: Chose Carbon `RegisterEventHotKey` over a `CGEventTap`/`NSEvent.addGlobalMonitorForEvents` for the global hotkey specifically to avoid an Accessibility/Input Monitoring permission prompt on first launch. The Settings-tab recorder re-registers by discarding and recreating the `HotKeyManager` instance (its `deinit` unregisters the old binding) rather than adding an update method to Carbon's registration.
- 2026-07-05: Overlay is an `NSPanel` subclass (`OverlayPanel`) overriding `canBecomeKey` to allow text editing in overlay-editable tabs, while `.nonactivatingPanel` keeps it from stealing focus/activating the app when just glancing at a read-only tab.
- 2026-07-05: User notes (`tabs.json`) are stored per-machine in Application Support, not committed to git — only the app's source syncs across laptops. Revisit if the user wants notes portable across machines too (would need a manual export/import or committing the JSON, since there's no network sync by design).
- 2026-07-05: Theme is a single 3-way choice (Light/Dark/Frosted Glass), not independent appearance+material toggles, per the user's phrasing ("theme to dark or bright or frosted glass"). Frosted Glass sets `NSApp.appearance = nil` (follows system); Light/Dark force `NSApp.appearance`.
- 2026-07-05: Frosted Glass was initially applied to both windows, then scoped to the overlay only after the user explicitly asked for the main editor window to stay normal/opaque. `AppDelegate.applyTheme` now always forces `mainWindow.isOpaque = true` regardless of theme; only `setOverlayTranslucent` responds to the Frosted Glass case. `EditorView` has no vibrancy background at all anymore.
- 2026-07-05: Settings is a pinned tab inside the same tab bar as content tabs (not a separate window or sidebar), per the user's framing of "2 tabs: one text-edit, one settings." It's excluded from the overlay's tab bar (`showsSettingsTab` defaults to `false`) since the overlay is meant for glancing at notes, not configuration. Later changed from a labeled pill inside the scrollable tab strip to an icon-only button pinned outside it, after the user reported it (and the "+") could get scrolled out of view.
- 2026-07-05: Content moved from a plain `String` to RTF-backed `NSAttributedString` (`TabItem.attributedContent`) to support the requested bold/font-size/font-family editing, using AppKit's own Font Panel (`NSFontManager.shared.orderFrontFontPanel`) rather than building a custom font picker — `NSTextView` already implements `changeFont(_:)` for rich text views, so this was far less code than a custom UI and gives full native font family/size/style control for free.
- 2026-07-05: Chose not to script real keyboard/mouse automation against the running app after an earlier attempt (in a prior session) clicked the wrong on-screen window due to a screen-coordinate mismatch. Verification in this session was limited to: build succeeds, app launches, screenshots confirm UI layout, and direct inspection/editing of the persisted JSON files. Anything requiring actual typing, hotkey presses, or feel/performance (like the tab-switch lag) needs the user to click through it.
- 2026-07-05: The reported ~1s lag switching tabs was diagnosed (not confirmed) as `.behindWindow` vibrancy blending on the *main* window under the Frosted Glass theme — that blend mode live-samples the entire desktop/other windows behind it on every redraw, which is expensive, whereas `.withinWindow` blends only within the window's own content and is cheap. Switched the main window to `.withinWindow`; left the overlay panel on `.behindWindow` since a small, short-lived HUD genuinely wants the true see-through glass look and the cost is bounded by how long it's shown. If the lag persists after this, look elsewhere (see Next steps).
